Planning · nothing built yet

Hit me — Growth Roadmap

Research and a phased plan for: a real database for multiple users, social-video export, a feedback channel, and making the app pay for itself — without over-engineering, and keeping the cost to each user very low.

The one idea to take away: the obvious build order (database → accounts → video) is backwards. The app already works offline on localStorage. So we ship the cheapest, demand-revealing things first (feedback + a tip jar) and only spend money once people show they want more.

01 Recommendations at a glance

Database

Supabase

Postgres + Auth + Row-Level Security. Real data integrity, security enforced inside the database, one vendor. Drops in behind the storage interface we already have.

Accounts

Supabase Auth

Free to 50k users. RLS does half the authorization for us. No signup needed to start — accounts are additive.

Video export

Browser → Remotion

Record in the browser for $0 to prove demand, then move to Remotion + AWS Lambda for polished MP4s at pennies each.

Feedback

In-app widget

One small form → one database table. Cheapest possible, and it ships first so it steers what we build next.

Money

Tip jar → Supporter

Start with a Ko-fi "buy me a coffee" link. Later a gentle ~$2–3/mo tier for storage & export. Core tool free forever.

Growth

The product itself

Shareable watermarked videos + SEO pages built from the lesson content already in the app. It markets itself.

02 Build order — by cost & risk

Each phase is gated on evidence from the one before it.

  • Phase 0 · Feedback widget + Ko-fi tip jar

    cost ~$0 · risk none

    Learn what to build and validate that people care — before spending anything.

  • Phase 1 · Accounts + cloud sync

    cost $0 → $25/mo (+ Vercel Pro $20 at monetisation) · risk low

    Supabase + RLS, only once people ask to sync songs across devices. One-time "import my local songs" on first sign-in.

  • Phase 2 · "Supporter" tier

    cost ~5% payment fee · risk low

    ~$2–3/mo (or a $20 lifetime): unlimited saves, export, cloud sync. Via a merchant-of-record (Lemon Squeezy/Paddle) so tax/VAT is handled for us.

  • Phase 3 · Social video — browser MVP

    cost ~$0 · risk medium

    Capture the chord animation + audio in the browser to a shareable, watermarked clip. Every share is an ad — the growth loop.

  • Phase 4 · Remotion + Lambda render

    cost pennies/video (+ check Remotion company licence) · risk medium

    Only if Phase 3 gets used: clean MP4s at any aspect ratio, rendered serverlessly at scale.

05 Where I pushed back on my own plan

  • Don't build accounts/video before demand. The tool works offline today; building persistence first would be over-engineering. → Feedback + tip jar lead.
  • Supabase's free tier sleeps after ~1 week idle. Bad for a quiet launch. → A daily Vercel Cron keep-alive ping, upgrade to Pro once traffic is steady.
  • Don't render video on a Vercel function (CPU/timeout/bandwidth). → Offload to Lambda; Vercel just starts the job and serves the link.
  • Remotion needs a paid licence for larger companies. → Browser MediaRecorder MVP first; it has no such string.
  • A stingy free tier kills word-of-mouth. → Set the free save cap high enough that only power users (the people happy to pay) ever meet the upgrade prompt.
  • Tax/VAT is a silent liability. → Use a merchant-of-record, not raw Stripe.

06 Your call

  1. Database: go with Supabase (my pick — integrity + RLS), or keep the already-scaffolded Firebase (zero migration)?
  2. Monetisation appetite: tip jar only, or tip jar → Supporter tier?
  3. Timing for accounts: build now, or wait until people ask to sync across devices?

Once you choose, the next step is to run brainstorming → spec → implementation plan on Phase 0/1 only — not the whole roadmap at once — and build from there.
